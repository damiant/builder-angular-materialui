# Card Components

## Overview

### Purpose

Use this component to generate card functionality

### When to Use

Card components are ideal for displaying related information and actions in a contained, visually distinct format. Use them when you need to:

- **Content organization** - Group related information, actions, or media into a cohesive, scannable unit
- **Dashboard layouts** - Create dashboard widgets that contain metrics, charts, or quick action sets
- **Product displays** - Showcase products, services, or portfolio items with images, descriptions, and actions
- **Article previews** - Display blog posts, news articles, or content summaries with preview images and metadata
- **User profiles** - Present user information, avatars, bio details, and profile actions in a contained format
- **Settings panels** - Organize configuration options, preferences, or account settings into logical groupings
- **Media galleries** - Display images, videos, or media content with titles, descriptions, and interaction options
- **List enhancements** - Elevate important list items or provide rich content alternatives to simple lists
- **Call-to-action sections** - Highlight important actions, announcements, or promotional content
- **Form sections** - Group related form fields and actions within visually distinct containers

### Architecture

**Moderate Coupling**: The Card component group consists of a primary container component (MatCard) that works together with several optional sub-components to create rich, structured content displays. The components are designed to work together harmoniously while maintaining flexibility for custom implementations.

### Component Interdependencies

The Card component group demonstrates moderate coupling through:

- **Structured composition**: MatCard serves as the primary container with optional sub-components for headers, content, actions, and media
- **Visual hierarchy**: Sub-components follow Material Design guidelines for proper spacing, typography, and visual hierarchy
- **Flexible layout**: Components can be combined in various ways while maintaining consistent styling and accessibility
- **Content organization**: Sub-components provide semantic structure for different types of card content (header, body, footer, actions)

## Components

### MatCard {Primary Component}

The primary container component that provides the foundational card structure with Material Design elevation, spacing, and visual treatment.

**Purpose:** Serves as the main container for card content, providing consistent styling, elevation effects, and responsive behavior while allowing flexible content composition through projection.

**Interface:**

```typescript
interface MatCardProps {
  appearance: "outlined" | "raised" | "filled"; // Visual appearance variant (default: 'raised')
  children: React.ReactNode; // Card content including optional sub-components
  class?: string; // Additional CSS classes for customization
  id?: string; // Unique identifier for the card
}

interface MatCardConfig {
  appearance?: "outlined" | "raised" | "filled"; // Default appearance for all cards
}
```

**Provides:** Base styling, elevation effects, responsive behavior, and content projection areas for sub-components

**Usage Example:**

```tsx
import { MatCard } from "@angular/material/card";

function BasicCardExample() {
  return (
    <mat-card appearance="raised">
      <mat-card-header>
        <mat-card-title>Card Title</mat-card-title>
        <mat-card-subtitle>Card subtitle</mat-card-subtitle>
      </mat-card-header>

      <mat-card-content>
        <p>
          This is the main content of the card where you can place any
          information.
        </p>
      </mat-card-content>

      <mat-card-actions>
        <button mat-button>Action 1</button>
        <button mat-button>Action 2</button>
      </mat-card-actions>
    </mat-card>
  );
}
```

### MatCardHeader {Layout Component}

A specialized container for card header content including titles, subtitles, and avatars with proper Material Design spacing and alignment.

**Purpose:** Provides structured layout for card header elements with consistent spacing, typography hierarchy, and avatar positioning according to Material Design specifications.

**Interface:**

```typescript
interface MatCardHeaderProps {
  children: React.ReactNode; // Header content including titles, subtitles, and avatars
  class?: string; // Additional CSS classes
}
```

**Context Dependencies:** Must be used within MatCard for proper styling and layout
**Parent Requirements:** MatCard container for consistent theming and spacing

**Usage Example:**

```tsx
<mat-card>

    <mat-card-title>Shiba Inu</mat-card-title>
    <mat-card-subtitle>Dog Breed</mat-card-subtitle>
  </mat-card-header>
</mat-card>
```

## Component Relationships

### Data Flow

Card components follow a simple parent-child data flow pattern where the MatCard container provides theming context and the sub-components receive content through projection. Data flows unidirectionally from parent components into card content areas.

### Context Sharing

Cards utilize Angular Material's theming system where the MatCard component provides theming context to all sub-components. This ensures consistent color palettes, typography, and elevation effects throughout the card structure.

### Event Handling

Event handling in card components follows standard Angular patterns where individual action buttons or interactive elements handle their own events. The card container itself does not intercept or modify events, allowing for flexible interaction patterns.

## Best Practices

### Usage Guidelines

- **Use cards for related content** - Group information, actions, and media that belong together conceptually to create coherent content units
- **Maintain consistent card sizes** - Within the same interface, use similar card dimensions and layouts to create visual harmony and improve scannability
- **Provide clear visual hierarchy** - Use card headers, titles, and content areas appropriately to guide user attention and comprehension
- **Include meaningful actions** - When using card actions, ensure they are relevant to the card content and provide clear value to users

### Accessibility

- **Semantic structure** - Card sub-components provide proper semantic HTML structure with appropriate heading levels and content organization
- **Keyboard navigation** - Interactive elements within cards are fully keyboard accessible with proper focus management and tab order
- **Screen reader support** - Card titles, subtitles, and content are properly structured for screen reader navigation and comprehension
- **Image accessibility** - Always provide descriptive alt text for card images and avatars to support assistive technologies

### Performance

- **Optimize card rendering** - Use OnPush change detection strategy for cards with static content to improve performance in large lists
- **Lazy load images** - For cards containing images, implement lazy loading to improve initial page load performance
- **Virtual scrolling** - When displaying many cards, consider using Angular CDK virtual scrolling to optimize rendering performance
- **Minimize DOM complexity** - Keep card content structure as simple as possible while maintaining functionality and accessibility
